### 2.2 強項

- **3 層架構清晰可維護** — Core → Daemon → Tray，依賴單向
- **Pure function 法官** — VerdictEngine 無副作用、可測試，這是架構亮點
- **11 種 Probe 覆蓋完整** — Wi-Fi 到 AI，Windows 特有檢測（註冊表代理檢測）
- **設計系統文件 (DESIGN.md)** — 角色定義、調色板、字體、元件規範，非常專業
- **自訂流程動畫** — Canvas-based 多目標流動可視化，獨特 UX
- **單例實例處理** — Mutex + EventWaitHandle 信號，成熟實作

### 2.3 債務

**P0（必須處理）**:
1. **MainWindow.xaml.cs ~1100 行** — 無 MVVM，全部 UI 邏輯在 code-behind。包含事件處理、狀態管理、動畫控制、分頁切換、設定對話框，無法單元測試、無法平行開發
2. **Polling-based 通訊** — TrayHost 和 MainWindow 各自獨立每 5 秒讀取 state.json。不是事件驅動，意味著：UI 延遲最少 5 秒、兩個消費者可能讀到不一致狀態、多餘 I/O

**P1（短期需解決）**:
3. **無 DI 容器** — 所有依賴手動 new，不利測試替換和生命週期
4. **無 CI 管線** — 沒有 .github/workflows。每次發布仰賴手動 publish.ps1
5. **--web 模式被 disable** — `CommandDispatcher` 回傳 error code 2
6. **Daemon/Tray 無測試** — 只有 Core 有單元測試
7. **Coverlet 已安裝但無 threshold/report** — 有覆蓋率工具但沒有效果

**P2（可延後）**:
8. **Web Dashboard 原始** — 單一 vanilla JS 頁面，無 Chart.js，只用 /api/state
9. **測試失敗未修** — DashboardMapperTests 因中英文 AlertsSummary 不一致而失敗
10. **部分中文字串硬編碼** — UiStrings 模式存在但未完全覆蓋
11. **state.json 與 samples.jsonl 雙存重複資料** — 資料一致性由寫入順序保障，無交易

## 3. 競品定位圖譜

我繪製了兩個維度的定位圖：

```
    高監控深度｜  PingPlotter ★              ┊
    （多跳、   ｜  ($39-299/yr)               ┊  NetStrata ★
     歷史趨勢）｜                              ┊  （分層判決）
              ｜                              ┊
              ｜  NextWatch                   ┊
              ｜  (.NET 8/WPF)                ┊
              ├──────────────────────────────┼──
              ｜  WinMTR (已死)               ┊  canireach ★ (靈感來源)
              ｜  Tracium ★                   ┊  Netok ★
    低監控深度｜  Qanary ★                    ┊  NetForge
              ｜  FluxNet ★                   ┊
              ｜  NetSpeedTray                ┊
              └──────────────────────────────┴──
                 純本地工具                      具分層判決
```

**關鍵發現**: 分層判決模型（Wi-Fi→LAN→ISP→海外→代理→AI）是 NetStrata 真正的護城河。在「分層判決 + 路徑可視化」這個交叉點上，**沒有任何競品同時做到這兩者**。PingPlotter 有路徑可視化但無層級判決；canireach 有層級判決但無路徑可視化。這個交叉點是 NetStrata 可以獨佔的位置。

### 各競品的獨特賣點 vs NetStrata 回應

| 競品 | 他們的賣點 | NetStrata 的回應 |
|------|-----------|-----------------|
| **PingPlotter** | 多跳歷史趨勢、AI 輔助、REST API、跨平台 | 免費 + 分層判決（PingPlotter 不做層級診斷） |
| **Tracium** | Rust/Tauri、<10MB、測速、bufferbloat、PDF 報告 | .NET 8 WPF 在 Windows 上更原生 |
| **Netok** | 網路診斷 + DNS 保護 + VPN 一條龍 | 工具範圍不同，NetStrata 純診斷 |
| **NetForge** | 交換器發現、CDP/LLDP、SSH 保險庫 | 定位為網路工程師工具箱，非健康監控 |
| **Qanary** | 即時連通性（~1s 反應）、紅綠燈 | NetStrata 如果改為事件驅動可追上 |
| **NextWatch** | SNMP、LAN 監控、Webhook | 補集關係——NextWatch 偏 LAN，NetStrata 偏 WAN |
| **FluxNet** | 頻寬監控、500ms 採樣、浮動 widget | 頻寬是 NetStrata 未覆蓋的維度 |

## 5. 功能性演進路線

```
Q3 2026                  Q4 2026                  Q1 2027
├──────────────────────┼──────────────────────┼───────────────────────┤

F1-F4 基礎體驗現代化 ──┤
 即時更新/Toast/Tray提示/自動啟動
                         F5-F8 核心護城河 ────┤
                          路徑可視化/單層鑽取/海外層強化/代理鏈可視化
                                                F9-F12 可訪問性 ────┤
                                                 Web儀表板/SSE/多通道警報/轉換感知
                                                                    F13-F16 歷史黏著度 ──
```

### Level 1：基礎體驗現代化（~1-2 週）

這些不是炫酷功能，而是「不做會扣分」的基本要求。

| # | 功能 | 具體實作 | 競品標竿 | 影響 |
|---|------|---------|---------|------|
| F1 | 事件驅動即時更新 | Daemon 透過命名管道推送 SampleCollectedEvent | Qanary ~1s vs 目前 5s | **每次使用都感受到的延遲消失** |
| F2 | Tray 懸停摘要 | 懸停顯示「正常｜延遲 15ms｜連接中…」 | 所有托盤工具 | 不用打開主視窗就能看到摘要 |
| F3 | Action Center Toast | 改用 Windows.UI.Notifications 留存通知 | Windows 原生標準 | 故障通知可回顧、不遺失 |
| F4 | 自動啟動預設啟用 | `--autostart` default true | NextWatch | 安裝後重啟自動恢復監控 |

### Level 2：核心診斷護城河（~2-3 週）

強化 NetStrata 獨有的分層診斷能力——這是競品沒有的差異點。

| # | 功能 | 具體實作 | 競品對照 | 獨佔性 |
|---|------|---------|---------|-------|
| F5 | 路徑逐跳可視化 | 現有 NetworkFlowControl 配上逐跳延遲圖 + 各跳統計 | PingPlotter（行業最佳） | **分層判決 + 路徑可視化 = 獨佔交叉點** |
| F6 | 單層鑽取 | 點擊任一層展開 raw probe 數據 + 歷史走勢 | canireach 只顯示 raw data | 讓 6 層判決可互動探索 |
| F7 | 海外層 GeoIP 強化 | 按大洲分組海外目標、獨立判斷各地連線 | Tracium 有世界地圖 | 分辨是「到 AWS 東京有問題還是整個海外都不行」 |
| F8 | 代理鏈可視化 | 繪製代理路徑圖（本地→HTTP代理→SOCKS→VPN→目標） | 無競品做到 | **完全獨佔**——這是 NetStrata 代理檢測深度的自然延伸 |

### Level 3：可訪問性（~2-3 週）

網路診斷工具的悖論：斷網時本地 UI 不受影響，但使用者無法在其他裝置上查看。

| # | 功能 | 具體實作 | 關鍵決策 |
|---|------|---------|---------|
| F9 | Web 儀表板（啟用 --web） | Kestrel minimal API + SSE + htmx SPA，預設 127.0.0.1 | 不需要 React——SSE + htmx 是完美配適 |
| F10 | SSE 即時串流 | `GET /api/events` 推送 SampleCollectedEvent | 瀏覽器無需輪詢 |
| F11 | 多通道警報 | Webhook (Slack/Discord/自訂) + 選配 Email | 轉換感知（transition-aware）不重複發 |
| F12 | 轉換感知警報 | 狀態變更才觸發，可設 repeat interval | 避免 alert fatigue |

### Level 4：歷史與趨勢（~2-3 週）

這是讓 NetStrata 從「實用工具」變成「持續使用工具」的關鍵。PingPlotter 願意賣 $299/yr 的核心價值就在這裡。

| # | 功能 | 具體實作 | 為什麼重要 |
|---|------|---------|-----------|
| F13 | SQLite 歷史資料庫 | Raw samples 保留 30 天 + 聚合資料無限期 | 解決「這週網路穩嗎？」——資料庫查詢取代 JSONL grep |
| F14 | 歷史走勢圖 | 1h/24h/7d/30d 延遲 + 抖動 + 遺失率 | 讓使用者看到模式（每天下午慢？週末才好？） |
| F15 | 每日/週報 | Markdown → PDF/HTML | ISP 投訴、工作日誌、自我檢視 |
| F16 | 穩定度評分（SQI） | 從歷史資料計算 Uptime% + 抖動 + 延遲百分位 → 0-100 | 一瞬間看懂今日網路品質 |

這些功能會讓 NetStrata 成為分層診斷領域的領導者。

| # | 功能 | 為何獨佔 | 技術可行性 |

## 6. 深入探討：本地落地的功能是否真實

這是整份報告中最重要的問題。我仔細思考後，回答是：

> **本地落地的功能是真實的，但有三個侷限。脫離了這三個侷限，本地功能就不夠真實。**

### 6.1 本地功能的真實價值

**1. 離線可靠** — 網路工具最大的矛盾是「斷網時用不了」。NetStrata 的本地架構在這個場景下反而是優勢——所有探測從本機發出，**斷網時仍然能執行 Wi-Fi / LAN 層檢測**。

**2. 即時零延遲** — 本地 Tray 圖示變化不需要經過伺服器、不需要載入。懸浮圖標的狀態提示比打開任何 App 都快。

**3. 隱私安全** — 不經第三方伺服器。適合企業環境、敏感網路。

**4. 系統整合深度** — WPF 的 NotifyIcon、ContextMenu、Taskbar integration 是 Web 技術無法取代的 Windows 原生體驗。Tray 是 Windows 使用者的核心 UX 模式。

**5. 無服務器依賴** — 不需要維護伺服器、不需要註冊帳號、不用擔心服務關閉。

### 6.2 本地功能的三個侷限（在哪裡不夠「真實」）

**侷限一：斷網時的通訊悖論**

*情境*：
```
你家網路斷了。你的桌機上 NetStrata 顯示「broadband: fail」。
但：誰能看到這個？你不在家。你家人打電話問你網路怎麼了。
你無法查看 NetStrata——你在手機上。
```

解決方案 → **F9（Web 儀表板）是打破此侷限的最低成本方式**。在同一個 LAN 上用手機訪問桌機的 Web 儀表板，不需要桌機能上網，只需要 LAN 暢通。

但更深層的問題：**如果 LAN 也斷了（例如路由器整個掛掉）**，Web 儀表板也無法訪問。這時唯一能通知你的是能繞過本地網路的路徑——也就是從桌機透過**另一個通道**發出的警報。

→ 這證明 **F11（多通道警報，特別是通過不同路徑）** 是必要的。例如桌機用手機熱點（如果 USB tethering 可用）發送警報，或利用 SMS/Email 透過蜂巢網路。

**侷限二：個人化單機**

*情境*：
```
你是 IT 工程師，管理 20 台機器。你想知道「所有機器的網路狀態」。
NetStrata 每台機器獨立安裝，你需要在 20 台機器上分別看。
```

解決方案 → **F18（多目標監控）** 在單機上監控多個外部目標可以部分解決。但真正的多機集中監控需要一個中央收集器架構——這是一個重大功能，屬於 Phase Z。

*輕量替代方案*：state.json 是純文字檔案，可以被 Prometheus exporter 或自訂腳本收集。在 **Level 3** 加入 REST API 後，甚至可以讓其他機器 pull。

**侷限三：本地文件難以分享**

*情境*：
```
你的 ISP 說「請提供網路不穩的證據」。你有 NetStrata 的 samples.jsonl。
但是無法給他們看——他們需要你截圖，或者發送原始 JSONL。
比較：PingPlotter 可以生成並分享報告連結。
```

解決方案 → **F15（報告匯出）** 和 **F21（一鍵分享）** 。

### 6.3 結論：本地功能的真實性評估

| 本地功能 | 真實嗎？ | 如果不全然，缺什麼？ | 補充方式 |
|---------|---------|------------------|---------|
| Tray 圖示狀態 | ✅ 高度真實 | — | — |
| Balloon 通知 | ✅ 但有限 | 無法留存回顧 | → Action Center Toast (F3) |
| WPF 主視窗 | ✅ 但有限 | 只能在本機操作 | → Web Dashboard (F9) |
| TUI/CLI | ✅ 真實 | — | — |
| state.json / samples.jsonl | ⚠️ 半真實 | 無法分享、無法查詢 | → SQLite (F13) + REST API |
| 警報通知 | ⚠️ 半真實 | 斷網時無法送達 | → 多通道警報 (F11) |
| 歷史資料 | ⚠️ 半真實 | JSONL 不可查、不可視覺化 | → SQLite (F13) + 趨勢圖 (F14) |

**核心洞察**：

> **讓本地功能「完全真實」的關鍵在於：建立一條繞過本機網路困境的通訊路徑。**

具體來說，這三件事構成「本地功能真實化三角形」：

```
              ┌───────────┐
              │ 本地核心    │  （Tray + WPF + CLI: 離線可靠、即時、深度整合）
              │ 診斷引擎    │
              └─────┬─────┘
                    │
        ┌───────────┼───────────┐
        │           │           │
        ▼           ▼           ▼
   ┌────────┐ ┌──────────┐ ┌──────────┐
   │LAN 通道 │ │替代路徑   │ │非即時分享 │
   │Web 儀表 │ │多通道警報 │ │報告匯出   │
   │板(SSE)  │ │(Webhook) │ │(PDF/URL) │
   └────────┘ └──────────┘ └──────────┘
   (同一LAN的    (繞過斷網    (離線產生的
    手機/平板)    的路徑)      可分享文檔)
```

只有三條腿都站穩，本地功能才是**真正真實的**。目前 NetStrata 只有第一條腿。

## 7. 戰略建議

### 7.1 不做的事（No-go）

1. **不要立即改為 Rust/Tauri** — 重寫成本極高，且放棄 WPF 在 Windows 上的原生優勢
2. **不要追 PingPlotter 的功能完整度** — PingPlotter 有 20 年累積、$299/yr 訂閱收入。用免費開源工具去追商業產品的功能集會分散焦點
3. **不要先做跨平台** — macOS/Linux 使用者可以先用 `--once` CLI（Core 已可編譯 net8.0 無 WPF），等 Windows 站穩再說
4. **不要先做 MAUI/手機 App** — 先做 Web 儀表板（F9），成本低 10 倍，覆蓋所有裝置

### 7.2 立即行動（Next 3 months 優先順序）

**Month 1 — 基礎現代化 + 事件驅動**:
```
W1:  事件匯流排實作（InProcessEventBus）
W2:  重構 TrayHost 為事件訂閱（移除 5s timer）
W3:  Action Center Toast + Tray 懸停摘要
W4:  重構 MainWindow 為 ViewModel（分批提取，不需一次到位）
```

驗收標準：state.json 輪詢在 TrayHost 中消除，→ Web Dashboard 可跨入下一步。

**Month 2 — Web 儀表板 + 護城河強化**:
```
W5-W6: Kestrel + SSE + htmx Web Dashboard（啟用 --web）
W7:    路徑逐跳可視化（現有 NetworkFlowControl 擴展）
W8:    GeoIP 海外層強化和代理鏈可視化
```

驗收標準：從手機瀏覽器（同 LAN）可看到即時診斷狀態。

**Month 3 — 警報 + 歷史**:
```
W9:   Webhook 多通道警報（Slack/Discord/自訂）
W10:  SQLite 歷史資料庫（保留策略 30天 raw + 永久聚合）
W11:  歷史趨勢圖（LiveCharts2） + 穩定度評分
W12:  Auto-update 整合 + CI 管線完成
```

驗收標準：使用者在 Discord 收到「連線恢復正常」通知。

### 7.3 一句話總結

> **NetStrata 的分層診斷模型是真正的護城河——但只有讓本地功能透過事件驅動+Web儀表板+多通道警報「走出去」，這個護城河才有意義。**
